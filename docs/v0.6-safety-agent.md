# v0.6 AI Content-Safety Agent — Design

> **Status: design draft.** No code shipped. This document fixes the
> shape of the feature so that when implementation begins, the
> architectural decisions are settled and the work decomposes
> cleanly into one feature dir + one service + one repo + a
> migration. Revise this doc rather than re-litigate the decisions
> ad hoc.

## 1. What the agent is, and isn't

The agent's job is to take a candidate video — a URL the parent
asked Curatables to consider, or a hit from a future live-search
feature — and decide whether it is safe to show a specific kid
profile, given that profile's safety criteria. It writes a
**verdict** to the database; it does not download videos, render
UI, or send notifications. Other layers handle those.

It is *not*:

- An online classifier in the request path. Verdicts are batch-
  produced by a queue worker, not synchronous on `/parent/add`.
- A blanket "is this video appropriate" oracle. The same video can
  be `confident_safe` for a 12-year-old's profile and
  `confident_unsafe` for a 5-year-old's; the criteria are per
  profile, evaluated once per (video, profile) pair.
- A replacement for parent review. `uncertain` verdicts go to a
  queue the parent reviews and decides; the agent is the first
  pass, not the only pass.
- A general LLM endpoint. Curatables is a self-hosted appliance;
  the agent runs against a local Ollama server on the same box, on
  CPU, against a 7B-class Q4 model. No cloud calls, no per-request
  cost.

## 2. Hardware envelope and what flows from it

The target box is a 6c/12t x86 mini-PC with
16-32 GB RAM, no GPU, passively or quietly cooled. That fixes:

- **Model**: tentatively Llama-3.1-8B-Instruct Q4_K_M via Ollama.
  ~5 GB on disk, ~6 GB resident. Other open-weights 7-8B Q4 models
  swap in via Ollama config — the agent code does not embed a
  model name.
- **Throughput**: a 7B Q4 on a 6c CPU runs ~8–15 tok/s. A typical
  verdict prompt (criteria + video metadata + JSON instruction)
  is ~600 input tokens and we cap output at ~150 tokens, so ~30 s
  per evaluation. **Performance budget: ~80–120 evaluations/hour
  in steady state on the 16 GB variant**, before thermal cooldown.
- **Concurrency**: one model instance, one verdict at a time. No
  attempt at parallel inference — that would compete with
  Curatables's HTTP serving threads on the same cores.
- **Thermal**: passively-cooled mini-PCs throttle at sustained
  100% CPU. The worker is **bursty by design** — see §6.

## 3. Data model

### 3.1 Per-profile safety criteria

Stored as a JSON blob on `profiles`, not a normalised table. The
shape evolves with prompt-engineering taste; a JSON column keeps
schema migrations cheap. The Pydantic-style validation lives in
`app/services/safety_agent.py`, not the DB.

Migration `app/db/migrations/NNNN_safety_criteria.sql` adds:

```sql
ALTER TABLE profiles ADD COLUMN safety_criteria TEXT;  -- JSON
ALTER TABLE profiles ADD COLUMN safety_threshold REAL DEFAULT 0.7;
```

`safety_criteria` shape (revisable):

```json
{
  "age_range": [4, 8],
  "topic_allow": ["nature", "science", "music", "how things work"],
  "topic_block": ["combat", "violence", "scary content", "ads"],
  "language": ["en", "pt"],
  "max_duration_seconds": 1200,
  "custom_prompt": "My kid is sensitive to loud noises and sudden cuts.",
  "block_live_streams": true
}
```

`safety_threshold` is the per-profile confidence floor an
auto-approval has to clear (0.7 = 70%). Tunable from the parent
profile-edit form.

### 3.2 Verdict log

A new table `safety_verdicts` records every (video, profile) the
agent evaluated. Append-only:

```sql
CREATE TABLE safety_verdicts (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  video_id      TEXT NOT NULL,         -- composite extractor_rawid
  profile_id    INTEGER NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  verdict       TEXT NOT NULL CHECK (verdict IN ('safe','uncertain','unsafe')),
  confidence    REAL NOT NULL,         -- 0.0–1.0 from the model
  rationale     TEXT NOT NULL,         -- single-paragraph explanation
  prompt_hash   TEXT NOT NULL,         -- SHA-256 of the rendered prompt
  model         TEXT NOT NULL,         -- e.g. "llama3.1:8b-instruct-q4_K_M"
  created_at    TEXT NOT NULL DEFAULT (datetime('now')),
  parent_action TEXT,                  -- 'approved', 'rejected', NULL = untouched
  parent_action_at TEXT
);
CREATE INDEX idx_safety_verdicts_video_profile
  ON safety_verdicts (video_id, profile_id, created_at DESC);
```

Storing `prompt_hash` and `model` makes it cheap to re-evaluate
when criteria change — old verdicts whose hash no longer matches
the rendered prompt are stale and the worker re-queues them.

### 3.3 Job queue

A new table `safety_jobs` for the worker to pull from:

```sql
CREATE TABLE safety_jobs (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  video_id    TEXT NOT NULL,
  profile_id  INTEGER NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  status      TEXT NOT NULL CHECK (status IN ('pending','running','done','failed')),
  attempts    INTEGER NOT NULL DEFAULT 0,
  last_error  TEXT,
  enqueued_at TEXT NOT NULL DEFAULT (datetime('now')),
  started_at  TEXT,
  finished_at TEXT,
  UNIQUE (video_id, profile_id, status)
);
```

The unique constraint dedups: enqueueing the same (video, profile)
twice while one is `pending` is a no-op. When a job finishes the
row is deleted, not retained — the verdict log (§3.2) is the
durable record.

## 4. Verdict types and what each one does

### 4.1 `confident_safe`

`verdict='safe' AND confidence >= profile.safety_threshold`.

The safety agent emits the verdict; the **content service**, when
it sees a verdict matching that condition for the active kid
profile, auto-approves the candidate and kicks off a download.
The kid sees the video on their grid the next time they reload.

The parent sees one summary line in `/parent/stats` per day:
"Safety agent auto-approved N videos for kid X, kid Y, ..." — no
per-video notification, because that defeats the point of the
batch path.

### 4.2 `uncertain`

`verdict='uncertain'`, OR `verdict='safe'` with low confidence,
OR `verdict='unsafe'` with low confidence.

The video lands on the parent's review queue at
`/parent/safety/queue` with the agent's rationale shown next to a
preview thumbnail. The parent picks **approve** (becomes a normal
add) or **reject** (deletes the candidate; no future ingestion of
that URL for that profile until the criteria change).

This is the path that catches genuinely ambiguous content — the
agent flags it and gets out of the way.

### 4.3 `confident_unsafe`

`verdict='unsafe' AND confidence >= profile.safety_threshold`.

The candidate is dropped, but **the verdict is reversible**. We
log it to `safety_verdicts` with `parent_action=NULL`, and the
parent sees a "Recently dropped by safety agent" view at
`/parent/safety/dropped` (paginated, last 200 entries). Each row
has a "Show me anyway" button that re-runs the add flow with the
verdict overridden — the agent's rationale is shown so the parent
makes an informed choice.

We do not silently delete data the parent might want; we
suppress it with full audit and one-click recovery.

## 5. The prompt and the JSON contract

The agent renders one prompt per (video, profile) pair. Inputs:

- Profile `safety_criteria` (the JSON blob).
- Video metadata: title, description, duration, channel, language,
  upload date, thumbnail URL, transcript snippets if available.
  **Not** the video file itself — the agent never sees pixels.
- A fixed system prompt that pins the JSON output schema.

Output schema (the model is asked for *exactly* this JSON):

```json
{
  "verdict": "safe" | "uncertain" | "unsafe",
  "confidence": 0.0,
  "rationale": "single-paragraph plain-English explanation",
  "matched_criteria": ["topic_block:violence", "age_range:above_max"]
}
```

Parsing rules:

- `confidence` clamped to [0.0, 1.0].
- `verdict` must be one of three; anything else → treated as
  `uncertain` with `confidence=0.0` and the parse error logged in
  `rationale`.
- `matched_criteria` is informational, not load-bearing. Used in
  the parent UI to highlight *why* something tripped.

Malformed JSON, hung Ollama, model OOM → see §7.

## 6. The worker

A long-running daemon thread inside the Curatables process —
**not** a separate service — driven by a SQLite poll loop. We do
not run it as systemd-timer-fired one-shots because:

- Spinning up a fresh Ollama context on every batch is wasteful;
  the model load is the expensive bit.
- Curatables already has lifespan hooks (the cache cleanup loop)
  using the same pattern.
- A failure in the worker thread does not take down the HTTP
  server — it logs, sleeps, retries.

### 6.1 Lifecycle

```
on app startup:
    if config.safety.enabled and ollama_reachable():
        start safety_worker_loop() as a daemon thread
```

If Ollama is unreachable at startup, the worker is **not** started
and `/healthz` reports `safety: "ollama_unreachable"`. The worker
re-checks reachability on each loop iteration and starts pulling
jobs if Ollama appears later.

### 6.2 Loop body

```
while not shutdown:
    if cooled_down_enough():
        job = next_pending_job()
        if job:
            run_one(job)
        else:
            sleep(idle_interval_seconds)
    else:
        sleep(thermal_cooldown_interval_seconds)
```

`cooled_down_enough()` reads `/sys/class/thermal/thermal_zone0/temp`
when present (Linux) and falls back to a rate limit (max N
verdicts per hour) when the thermal sensor isn't available.
Defaults: cooldown when zone temp > 75 °C, target ≤ 60 °C; rate
limit 60/hour. All three numbers are in `config.safety`.

### 6.3 Jobs are enqueued by

- The `add` flow in `parent_content` when the parent submits a URL
  *and* one or more profiles have `safety_criteria` set. One job
  per (video, profile) pair.
- A `recompute` admin action when the parent edits a profile's
  criteria — the job enqueues all videos visible to that profile.
- (Future) The live-search feature, when implemented.

## 7. Failure modes and retry policy

| Failure                                | Retry | Verdict on give-up | Notes |
|----------------------------------------|-------|--------------------|-------|
| Ollama unreachable                     | yes, with exponential backoff up to 5 min | none — job stays `pending` | Worker pauses the whole queue; resumes when Ollama answers a `/api/tags` ping. |
| Model OOM (Ollama returns 5xx)         | yes, ×3 with 30 s delay | `uncertain`, conf 0.0 | Surfaces as "evaluator transient failure" in the rationale. |
| Malformed JSON output                  | yes, ×2 with the same prompt | `uncertain`, conf 0.0 | Most "malformed" responses are model brain farts that retry fixes. |
| HTTP timeout                           | yes, ×3 with 30 s delay | `uncertain`, conf 0.0 | 60 s per attempt timeout. |
| Profile deleted mid-flight             | no | n/a, job dropped | `ON DELETE CASCADE` cleans up. |
| Ollama returns persistently bad output | mark job `failed` after attempt 5 | `uncertain` written | `last_error` carries the diagnostic. |

The recurring theme: **fail safe to `uncertain`**. If the agent
cannot decide, the parent decides. We never silently flip a
candidate to `safe` because of an evaluator failure.

## 8. Parent UX surfaces

Three new pages, all under `/parent/safety/`:

- **`/parent/safety/queue`** — the `uncertain` review queue, like
  the existing `/parent/content` review surface but with the
  agent's rationale and matched-criteria badges shown on each row.
- **`/parent/safety/dropped`** — the recoverable `confident_unsafe`
  log; "Show me anyway" reopens the add flow.
- **`/parent/safety/settings`** (new tab on profile edit) — per-
  profile criteria editor: age sliders, allow/block topic chips,
  language multi-select, duration cap, free-text custom prompt,
  confidence threshold slider.

A small status chip in the parent topnav reports queue depth and
worker health: green = idle, amber = N pending, red = Ollama
unreachable. Click takes you to `/parent/safety/queue`.

## 9. Code layout

Following the existing routes → services → repositories rule:

```
app/
├── features/parent_safety/
│   ├── __init__.py
│   └── router.py                 # /parent/safety/* routes
├── services/
│   ├── safety_agent.py           # prompt build, verdict parse, Ollama call
│   └── safety_worker.py          # daemon loop, thermal/rate gating
├── repositories/
│   └── safety_repo.py            # CRUD on safety_jobs + safety_verdicts
└── db/migrations/
    └── NNNN_safety_agent.sql     # tables + profile column adds
```

Lifespan registration in `app/main.py`: start `safety_worker.run()`
in a daemon thread when `config.safety.enabled`. Cancellation on
shutdown follows the cache-cleanup-loop pattern.

## 10. Open questions

These are still open. Land them in a follow-up section or a PR
discussion before implementation:

- **Transcript ingestion.** Curatables already downloads subtitles
  when the source has them. Feeding the first ~2 KB of subtitles
  into the prompt is high-signal but expands the prompt size and
  thus latency. Ship without and add later? Ship with a config
  cap?
- **Model choice.** Llama-3.1-8B-Instruct Q4_K_M is the default,
  but Phi-3-mini (3.8B Q4) is fast enough that a 32 GB box could
  run it in parallel with Ollama answering other tools. Worth a
  config knob for "prefer speed vs. quality."
- **Re-evaluation cadence.** When a model in Ollama is updated,
  every old verdict's `prompt_hash` is still valid (the prompt is
  the same), but the model identifier in `safety_verdicts.model`
  changes. Do we mass-re-queue, or only re-queue on demand?
- **Per-criterion confidence vs. overall confidence.** The current
  schema gives one `confidence` for the whole verdict. A future
  evolution could ask the model for per-criterion scores so the
  parent UI can show "100% confident on age, 60% confident on
  topic." Out of scope for v0.6.
- **Live-search integration.** Mentioned in the README roadmap;
  the agent design above already supports it (just enqueue jobs
  for hits). The search feature itself is separate work.

## 11. Implementation slices

Suggested ordering for landing the feature, each as its own PR:

1. Migration + repo (`safety_jobs`, `safety_verdicts`,
   profile columns). Tests.
2. `safety_agent.py` service: prompt builder, JSON parser, Ollama
   client. Unit tests with a stubbed HTTP client.
3. `safety_worker.py` worker loop with thermal gating. Tests.
4. `parent_safety` feature dir: queue, dropped, settings routes
   and templates.
5. Lifespan wiring in `app/main.py` + topnav status chip.
6. Live-search integration (waits on the search feature).

Anything before slice 4 is invisible to users; slice 4 is when the
feature becomes usable end-to-end.
